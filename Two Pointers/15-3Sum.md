# 15. 3Sum

## Problem

Given an integer array `nums`, return all **unique** triplets `[nums[i], nums[j], nums[k]]` where the three values sum to zero. The same element can't be reused within a triplet, and the result must contain no duplicate triplets.

Concrete example:

```
Input:  nums = [-1, 0, 1, 2, -1, -4]
Sorted: [-4, -1, -1, 0, 1, 2]
Output: [[-1, -1, 2], [-1, 0, 1]]
```

Note the output is **values, not indices**, and `[-1, 0, 1]` appears only once even though two different `-1`s could form it.

---

# My Learning Journey

this was a long grind — the algorithm itself is "two sum II run n times," but almost every bug from problems 125 and 167 resurfaced here, plus a brand-new duplicate-handling layer that's the real reason 3Sum is a Medium.

it started with a conceptual block: i didn't want to sort the array because i thought it would violate the O(1) space recommendation. that was the wrong mental model — i learned that for 3Sum the accepted answer *is* sort + two pointers, the sort's O(n log n) is dominated by the O(n²) scan, and by convention the sort space isn't counted. the O(1)-no-sort instinct came from 167, where the array is *handed to you sorted*. 3Sum doesn't do that, so the whole pattern is "array isn't sorted but two pointers would help → sort it first."

once i accepted the sort, the bugs came in waves, and nearly every one was a repeat of an earlier lesson:
- `nums = nums.sort()` returned `None` (the immutability cousin of `s.lower()` from 125 — `.sort()` mutates in place and returns nothing).
- `l = 0` instead of `l = i+1` (left pointer overlapping the fixed element).
- dead guard clauses `(i != l or i != r)` that contaminated the branch logic.
- no pointer movement after appending → infinite loop (in 167 i `return`ed here, so it never came up).

then the genuinely new part: duplicate handling. i got stuck on two things and needed them broken all the way down:
1. **why `continue` and not `i += 1`** — i learned you can't steer a `for ... in range()` variable by hand; the range reassigns it every pass, so skipping an iteration requires the control-flow keyword `continue`, not arithmetic.
2. **why we don't dedup `r`** — i learned that once `i` and `l` are fixed, `r` is *forced* to one value (`-(nums[i]+nums[l])`), so it has no freedom to independently create a duplicate. dedup the two free dials (`i` and `l`); the forced dial (`r`) is automatic.

the "3-dial lock" model is what finally made the dedup click: police the dials you can turn freely, ignore the one that's pinned.

# Accepted Solution(s)

```python
class Solution:
    def threeSum(self, nums: List[int]) -> List[List[int]]:
        trip = []
        nums.sort()                                  # sort enables two-pointer direction
        for i in range(len(nums)):
            if i > 0 and nums[i] == nums[i-1]:       # skip duplicate FIRST element
                continue
            l, r = i+1, len(nums)-1                   # only look forward of i
            while l < r:
                if nums[i]+nums[l]+nums[r] > 0:       # too big → shrink → move r
                    r -= 1
                elif nums[i]+nums[l]+nums[r] < 0:     # too small → grow → move l
                    l += 1
                else:                                 # == 0 → record triplet
                    trip.append([nums[i], nums[l], nums[r]])
                    l += 1
                    r -= 1
                    while l < r and nums[l] == nums[l-1]:  # skip duplicate LEFT values
                        l += 1
        return trip
```

---

# Why It Works

3Sum is **Two Sum II run n times.** Fix one number `nums[i]`, and finding the other two reduces to: *find a pair in the remaining array that sums to `-nums[i]`* — exactly problem 167 with a computed target. The sort is what makes the inner search an O(n) two-pointer scan instead of an O(n²) brute force.

**The invariant:** at each `i`, every element before `i` has already been fully explored as a first element, so the inner scan only looks *forward* (`l = i+1`). That's why no triplet is double-counted from the structure, and why the dedup skips are the only extra machinery needed.

---

# Traversal Logic

How we move through the data — an outer loop plus a data-driven two-pointer scan.

```
sorted: [-4, -1, -1, 0, 1, 2]
          i    →                       outer loop fixes the first element
              l →            ← r        inner pointers converge on the rest

For each i:  l starts at i+1, r at the end.
sum > 0 → move r left   (shrink)
sum < 0 → move l right  (grow)
sum == 0 → record, move BOTH inward
```

Same data-driven movement as 167 (the sum decides who moves), wrapped in an outer loop over the fixed element.

---

# Business Logic

The decisions made at each step — separate from how we move.

- compute `nums[i] + nums[l] + nums[r]`
- `> 0` → move `r` (traversal consequence of a business comparison)
- `< 0` → move `l`
- `== 0` → append `[nums[i], nums[l], nums[r]]`
- **dedup decisions:** skip this `i` if it repeats the previous value; after a match, skip `l` past repeated values

The comparisons and the dedup checks are business. The pointer moves are traversal.

# Variable Roles

| Variable | Represents | Stores | How It Changes | Why It Exists |
|----------|------------|--------|----------------|---------------|
| `i` | index of the fixed first element | int index | driven by `range`; skipped via `continue` on dup | anchors the triplet; reduces 3Sum to 2Sum |
| `l` | left pointer into the remainder | int index | `+= 1` when sum small or skipping dups | front of the inner search window |
| `r` | right pointer into the remainder | int index | `-= 1` when sum big or after a match | back of the inner search window; value forced once i,l fixed |
| `trip` | the result | list of `[int,int,int]` | appended on each match | collects the unique triplets |

# Visual Walkthrough

Input: `nums = [-1, 0, 1, 2, -1, -4]` → sorted `[-4, -1, -1, 0, 1, 2]`

| i | nums[i] | l | r | sum | Action |
|---|---------|---|---|-----|--------|
| 0 | -4 | 1→4 | 5 | <0 throughout | l walks up, no triplet |
| 1 | -1 | 2 | 5 | -1-1+2=0 | append [-1,-1,2], l→3, r→4 |
| 1 | -1 | 3 | 4 | -1+0+1=0 | append [-1,0,1], l→4, r→3, stop |
| 2 | -1 | — | — | — | nums[2]==nums[1] → **continue** (kills the dup) |
| 3 | 0 | 4 | 5 | 0+1+2=3>0 | r→4, l==r, stop |
| 4,5 | — | — | — | — | l starts ≥ r, inner loop skipped |

Result: `[[-1,-1,2],[-1,0,1]]` — no duplicates.

ASCII of the dedup-`i` moment:

```
[-4, -1, -1, 0, 1, 2]
      ↑    ↑
    i=1   i=2   same value (-1) → i=2 does continue, skipped entirely
```

# Optimization Journey

1. **Brute Force:** three nested loops over all index triples. O(n³) time, O(1) space.
2. **Bottleneck:** the O(n³) — re-checking pairs the sorted order could rule out.
3. **Observation:** if the array is sorted, fixing one element turns the rest into 2Sum II, solvable with an O(n) two-pointer scan.
4. **Optimization:** sort, then outer loop + inner two pointers → O(n²). Add dedup skips for uniqueness.
5. **Final Approach:** the accepted solution. An interviewer discovers it by asking "can I reduce this to a problem I already know?" — fix one, 2-sum the rest.

# Debugging Lessons

The big realizations:

- **`continue` vs `i += 1`:** you can't advance a `for ... in range()` loop variable by hand — the range reassigns it each pass. Skipping an iteration is control flow, so it needs `continue`. `i += 1` is arithmetic that evaporates and doesn't stop the body.
- **why no `r` dedup:** once `i` and `l` are fixed, `r` is forced to `-(nums[i]+nums[l])` — a single value with no freedom. It can't independently create a duplicate, so deduping the two *free* dials (`i`, `l`) covers it automatically.
- **indent encodes logic:** nesting the scan inside the dup-`if` made it run only on duplicates. The guard must `continue` and the scan must live at the loop's own indent level.
- repeat lessons from 125/167: immutability (`.sort()` returns None), pointer start position (`l=i+1`), `if/elif` structure, bounds-check-before-index in the dup-`l` while.

# NeetCode / Official Solution

## Code

```python
class Solution:
    def threeSum(self, nums: List[int]) -> List[List[int]]:
        res = []
        nums.sort()
        for i, a in enumerate(nums):
            if i > 0 and a == nums[i - 1]:
                continue
            l, r = i + 1, len(nums) - 1
            while l < r:
                threeSum = a + nums[l] + nums[r]
                if threeSum > 0:
                    r -= 1
                elif threeSum < 0:
                    l += 1
                else:
                    res.append([a, nums[l], nums[r]])
                    l += 1
                    while nums[l] == nums[l - 1] and l < r:
                        l += 1
        return res
```

## Comparison to My Solution

| | NeetCode | Mine |
|---|----------|------|
| Algorithm | sort + two pointers + dedup | identical |
| Index/value | `enumerate` → `a = nums[i]` | `nums[i]` directly |
| Sum stored | `threeSum` variable | recomputed in branch checks |
| Dup-`l` guard order | `nums[l] == nums[l-1] and l < r` | `l < r and nums[l] == nums[l-1]` |
| `r` dedup | none (forced-dial) | none (forced-dial) |
| Time / Space | O(n²) / O(1) | O(n²) / O(1) |

The algorithm is the same. Two cosmetic differences: NeetCode uses `enumerate` to get `a = nums[i]` for cleaner reads, and stores the sum in a variable. One subtle difference: NeetCode's dup-`l` while checks `nums[l] == nums[l-1]` *before* `l < r`, whereas mine guards `l < r` first — mine is the more defensive ordering (short-circuit protects the index access), consistent with the bounds-check-first lesson from Valid Palindrome. NeetCode's happens to be safe here due to how the bounds work out, but mine is correct-by-construction.

# Deep Understanding

The non-obvious core is the **forced-dial argument** for why `r` needs no dedup. A triplet is a 3-dial lock that must sum to zero. Dials `i` and `l` you turn freely, so each can land on a repeated value and produce a duplicate — both must be deduped. But once `i` and `l` are set, `r` is pinned to exactly `-(nums[i]+nums[l])`. A pinned dial has no independent choice, so it cannot generate a new duplicate combination. Concretely, when you skip duplicate `l` values after a match, the `l`/`r` pair moves together and `l` skipping past repeats ends the relevant scan before `r` could revisit a duplicate-producing value. Deduping one free dial covers the forced one.

The second non-obvious bit is the `for`-loop variable: it is *owned by the range*, not by your code. This is why `i += 1` is inert and `continue` is required — a lesson about Python loop semantics, not about 3Sum specifically.

# Interview Explanation (2 Minutes)

The signals are "triplets summing to zero" plus "no duplicates," which points at sort + two pointers. After sorting, I fix the first element at index `i`, then the problem becomes finding two numbers in the remainder that sum to `-nums[i]` — that's Two Sum II, an O(n) two-pointer scan, because the array is sorted.

For each `i` I set `l = i+1` and `r = len-1`, and move based on the sum: too big, drop `r`; too small, advance `l`; zero, record the triplet and move both inward.

Duplicates are the tricky part. They come from repeated *values* at the free positions. If the current `i`'s value equals the previous one, every triplet starting with it is already found, so I `continue`. After recording a match, I skip `l` past any repeats of the value I just used. I don't dedup `r` separately, because once `i` and `l` are fixed, `r` is forced to one value — it has no freedom to create a new duplicate.

Complexity is O(n²): sort is O(n log n) but dominated by the n outer iterations times the O(n) inner scan. Space is O(1) excluding the output. The main tradeoff versus a hash-set approach is that sorting gives cleaner dedup and O(1) space instead of O(n).

# Common Follow-Up Questions

- **"How do you avoid duplicate triplets?"** → the #1 follow-up. Dedup the free dials: skip `i` when it repeats the previous value (`continue`), skip `l` past repeats after a match. `r` is forced, so it's automatic.
- **"Why doesn't `r` need a dedup check?"** → once `i` and `l` are fixed, `r` is pinned to `-(nums[i]+nums[l])`; no independent choice, no new duplicate.
- **"What's the complexity and why?"** → O(n²); sort is dominated by the nested scan.
- **"Extend to 4Sum?"** → add one more outer loop fixing a second element, then two-pointer the rest → O(n³).
- **"Could you do it without sorting?"** → hash set, O(n²) time but O(n) space and messier dedup; sorting is preferred.

# Key Takeaways

- 3Sum = sort + (Two Sum II run n times). Reduce hard problems to ones you already know.
- when an unsorted array would benefit from two pointers, **sort it first** — sorting cost is often dominated anyway.
- dedup the *free* positions (`i`, `l`); the *forced* position (`r`) takes care of itself.
- you can't advance a `for ... in range()` loop by mutating its variable — use `continue` to skip, `break` to stop.

# Final Mental Model

Picture lining everyone up by height (the sort). You walk down the line and pick each person as the "anchor." For that anchor, two helpers start just past them — one at the near end, one at the far end — and walk toward each other looking for a pair whose heights, plus the anchor's, balance to zero. If two anchors are the same height, you don't redo the search — you've already found everything that anchor can make. And you never separately police the far helper, because once the anchor and near helper are chosen, the far one's required height is fixed — there's only one person who fits, no room for a repeat.
